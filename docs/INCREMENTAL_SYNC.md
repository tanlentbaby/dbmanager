# 增量同步和断点续传使用指南

**版本：** v1.8.0+  
**最后更新：** 2026-03-20

---

## 📈 增量同步

### 使用场景

- **定期同步** - 每天/每小时同步新增数据
- **实时同步** - 持续同步最新数据
- **数据备份** - 增量备份减少时间
- **数据分发** - 分发新增数据到多个目标

### 同步原理

**基于 ID 的增量：**
```sql
SELECT * FROM table WHERE id > last_sync_id
```

**基于时间戳的增量：**
```sql
SELECT * FROM table 
WHERE updated_at > last_sync_time
  AND id > last_sync_id
```

### 同步步骤

**1. 配置选项**
1. 导航到"增量同步"页面
2. 配置同步选项
   - 使用时间戳过滤
   - 检测删除记录
   - 批量大小

**2. 生成同步计划**
1. 点击"生成同步计划"
2. 查看新增/更新/删除记录数
3. 评估同步时间

**3. 执行同步**
1. 确认同步计划
2. 点击"开始同步"
3. 实时查看进度

**4. 导出报告**
- 导出同步报告
- 记录同步结果
- 用于审计和验证

### 同步选项

| 选项 | 说明 | 推荐 |
|------|------|------|
| 使用时间戳 | 同时过滤 ID 和时间 | 开启 |
| 检测删除 | 检测源端删除记录 | 按需 |
| 批量大小 | 每批处理行数 | 1000-5000 |

## ⏯️ 断点续传

### 使用场景

- **大表传输** - 百万/千万级数据
- **网络不稳定** - 可能中断的场景
- **定时任务** - 分多次完成传输
- **资源限制** - 避免长时间占用资源

### 续传原理

**检查点机制：**
```json
{
  "id": "transfer_123",
  "tableName": "users",
  "lastProcessedId": 50000,
  "processedRows": 50000,
  "totalRows": 100000,
  "status": "paused"
}
```

**续传流程：**
1. 保存检查点
2. 中断传输
3. 加载检查点
4. 从断点继续

### 使用步骤

**1. 开始传输**
1. 导航到"断点续传"页面
2. 指定表名
3. 点击"开始传输"

**2. 暂停传输**
1. 点击"暂停"按钮
2. 检查点自动保存
3. 可以随时恢复

**3. 恢复传输**
1. 点击"续传"按钮
2. 从断点继续
3. 查看剩余时间

### 进度信息

| 信息 | 说明 |
|------|------|
| 总行数 | 需要传输的总行数 |
| 已处理 | 已完成的行数 |
| 处理速度 | 每秒处理行数 |
| 预计剩余 | 预计完成时间 |
| 最后 ID | 最后处理的 ID |

## 💡 使用技巧

### 增量同步技巧

**首次全量：**
1. 先执行全量同步
2. 建立检查点
3. 后续增量同步

**定期同步：**
1. 设置定时任务
2. 每天/每小时执行
3. 监控同步延迟

**性能优化：**
1. 使用索引列过滤
2. 增大批量大小
3. 并行同步多表

### 断点续传技巧

**大表传输：**
1. 分批传输
2. 每批 10-50 万行
3. 避免超时

**网络不稳定：**
1. 启用自动重试
2. 减小批量大小
3. 频繁保存检查点

**资源限制：**
1. 夜间传输
2. 限速传输
3. 分多天完成

## ⚠️ 注意事项

### 性能建议

**增量同步：**
- 建议 < 100 万行/次
- 使用索引列过滤
- 避免业务高峰期

**断点续传：**
- 每批 < 50 万行
- 检查点持久化
- 监控传输速度

### 安全建议

**增量同步：**
- 只读权限同步
- 验证数据一致性
- 记录同步日志

**断点续传：**
- 检查点加密存储
- 验证检查点完整性
- 防止重复传输

## 🐛 常见问题

### Q: 增量同步漏数据？

**A:** 检查：
1. ID 是否连续
2. 时间戳是否准确
3. 检查点是否正确保存

### Q: 续传失败？

**A:** 处理：
1. 检查检查点文件
2. 验证检查点完整性
3. 重新生成检查点

### Q: 同步速度慢？

**A:** 优化：
1. 增大批量大小
2. 使用索引列
3. 并行同步
